# CCCP Advisory System

Локальная RAG-система (Retrieval-Augmented Generation) для ответов на вопросы по содержимому базы знаний. Система принимает `.md`-файлы, индексирует их в векторной базе данных и отвечает на вопросы пользователей, используя только информацию из загруженных документов.

## Быстрый старт

```bash
docker-compose up --build
```

При первом запуске Ollama скачает необходимые модели (~5 ГБ). Дождитесь сообщения `All models are ready.` в логах контейнера `ollama`, после чего система готова к работе.

## Сервисы

| Сервис   | Порт  | Назначение                          |
|----------|-------|-------------------------------------|
| admin    | 8001  | Загрузка и индексация базы знаний   |
| query    | 8000  | Ответы на вопросы пользователей     |
| qdrant   | 6333  | Векторная база данных               |
| ollama   | 11434 | Inference-сервер (embeddings + LLM) |

## API

### POST /ingest

**Сервис:** admin (порт 8001)

Читает все `.md`-файлы из директории `admin/knowledge/`, разбивает на чанки, генерирует эмбеддинги и загружает в Qdrant. Вызывается автоматически при старте сервиса, но может быть вызван вручную после добавления новых файлов.

> Каждый вызов полностью пересоздаёт коллекцию в Qdrant — все предыдущие данные удаляются и заменяются новыми.

**Запрос:**

```bash
curl -X POST http://localhost:8001/ingest
```

Тело запроса не требуется.

**Ответ (200):**

```json
{
  "status": "ok",
  "chunks_count": 42
}
```

| Поле           | Тип    | Описание                                  |
|----------------|--------|-------------------------------------------|
| `status`       | string | Статус операции (`"ok"`)                  |
| `chunks_count` | int    | Количество проиндексированных чанков      |

**Ошибки:**

| Код | Причина                                                  |
|-----|----------------------------------------------------------|
| 500 | Директория `knowledge/` не найдена, нет `.md`-файлов, Qdrant или Ollama недоступны |

---

### POST /ask

**Сервис:** query (порт 8000)

Принимает вопрос на естественном языке и возвращает ответ, сгенерированный на основе релевантных фрагментов из базы знаний.

**Запрос:**

```bash
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "Какие продукты вы предлагаете?"}'
```

| Поле       | Тип    | Обязательное | Описание          |
|------------|--------|--------------|-------------------|
| `question` | string | да           | Вопрос пользователя |

**Ответ (200):**

```json
{
  "answer": "Согласно информации из базы знаний, мы предлагаем..."
}
```

| Поле     | Тип    | Описание                                           |
|----------|--------|----------------------------------------------------|
| `answer` | string | Сгенерированный ответ или `"Информация не найдена"` |

**Ошибки:**

| Код | Причина                                                        |
|-----|----------------------------------------------------------------|
| 400 | Проблемы с кодировкой запроса (не удалось декодировать тело)   |
| 500 | Qdrant или Ollama недоступны, ошибка при генерации ответа      |

Сервис автоматически пытается перекодировать тело запроса из windows-1251, cp866, koi8-r и iso-8859-5 в UTF-8, если исходная кодировка не UTF-8.

## Алгоритм RAG

RAG (Retrieval-Augmented Generation) — подход, при котором ответ языковой модели основывается не на её внутренних знаниях, а на конкретных документах, найденных по запросу. Это позволяет получать точные ответы, привязанные к актуальному содержимому базы знаний, и минимизирует галлюцинации модели.

Алгоритм состоит из двух фаз: **индексация** (однократная подготовка данных) и **поиск с генерацией** (при каждом вопросе).

### Фаза 1: Индексация (Ingestion)

Цель — преобразовать текстовые документы в векторные представления и сохранить их в базе данных для быстрого семантического поиска.

```
.md файлы -> Чанкинг -> Эмбеддинги -> Qdrant
```

#### Шаг 1. Чтение документов

Система читает все файлы с расширением `.md` из директории `admin/knowledge/`. Файлы считываются в кодировке UTF-8 и сортируются по имени для детерминированного порядка обработки.

#### Шаг 2. Разбиение на чанки (Chunking)

Каждый документ разбивается на небольшие фрагменты (чанки) с помощью `RecursiveCharacterTextSplitter` из LangChain. Алгоритм разбиения:

- **Размер чанка:** 300 символов. Это достаточно мало, чтобы каждый чанк содержал одну смысловую единицу, и достаточно велико, чтобы сохранить контекст.
- **Перекрытие (overlap):** 100 символов. Соседние чанки перекрываются на 100 символов, чтобы информация, оказавшаяся на границе разбиения, не потерялась.
- **Рекурсивная стратегия:** сплиттер сначала пытается разделить текст по двойным переносам строк (`\n\n`), затем по одинарным (`\n`), затем по пробелам, и только в крайнем случае — посимвольно. Это сохраняет естественную структуру текста (абзацы, предложения).

Пример: документ из 1000 символов будет разбит примерно на 5 чанков, где каждый чанк содержит ~300 символов, а каждые два соседних чанка имеют ~100 общих символов.

#### Шаг 3. Генерация эмбеддингов

Каждый чанк преобразуется в числовой вектор (эмбеддинг) с помощью модели **BGE-M3**, запущенной в Ollama. BGE-M3 — мультиязычная модель, поддерживающая русский язык.

Эмбеддинг — это вектор фиксированной размерности, в котором семантически близкие тексты располагаются рядом друг с другом в векторном пространстве. Например, эмбеддинги фраз «цена продукта» и «стоимость товара» будут находиться близко, несмотря на разные слова.

Все чанки отправляются на генерацию эмбеддингов одним батчем.

#### Шаг 4. Сохранение в Qdrant

Перед загрузкой новых данных коллекция `info_chunks` в Qdrant полностью удаляется и создаётся заново с параметрами:

- **Размер вектора:** определяется автоматически по размерности первого эмбеддинга.
- **Метрика расстояния:** косинусное сходство (Cosine).

Каждый чанк сохраняется как точка (point) с:
- **id** — порядковый номер чанка (0, 1, 2, ...).
- **vector** — эмбеддинг чанка.
- **payload** — `{"text": "..."}` — исходный текст чанка для последующего извлечения.

### Фаза 2: Поиск и генерация ответа (Retrieval + Generation)

Цель — найти наиболее релевантные фрагменты базы знаний и сгенерировать ответ на их основе.

```
Вопрос -> Эмбеддинг -> Поиск в Qdrant -> Контекст + Промпт -> LLM -> Ответ
```

#### Шаг 1. Эмбеддинг вопроса

Вопрос пользователя преобразуется в вектор той же моделью **BGE-M3**, что использовалась при индексации. Это критически важно: поиск работает корректно только если вопрос и документы закодированы одной и той же моделью.

#### Шаг 2. Семантический поиск

Вектор вопроса сравнивается со всеми векторами чанков в Qdrant. Поиск выполняется по косинусному сходству (от 0 до 1, где 1 — полное совпадение).

Параметры поиска:
- **top_k = 3** — возвращаются не более 3 наиболее похожих чанков.
- **similarity_threshold = 0.3** — чанки с косинусным сходством ниже 0.3 отбрасываются как нерелевантные.

Если ни один чанк не прошёл порог, система сразу возвращает `"Информация не найдена"`, не обращаясь к LLM.

#### Шаг 3. Формирование промпта

Тексты найденных чанков объединяются в один блок контекста (разделитель — двойной перенос строки). Затем формируются два сообщения для LLM:

**Системный промпт** задаёт роль и ограничения:
> Ты — полезный ИИ-ассистент. Отвечай на вопросы пользователя, используя ТОЛЬКО предоставленный контекст. Если в контексте недостаточно информации для ответа, ответь строго: "Информация не найдена".

**Пользовательское сообщение** содержит контекст и вопрос:
```
Контекст:
{тексты найденных чанков}

Вопрос: {вопрос пользователя}

Ответь на вопрос, используя только информацию из контекста выше.
```

Двойное указание ограничения (в системном промпте и в пользовательском сообщении) снижает вероятность того, что модель будет отвечать на основе собственных знаний.

#### Шаг 4. Генерация ответа

Промпт отправляется в модель **Qwen 2.5 7B**, запущенную в Ollama. Модель генерирует ответ, основываясь исключительно на предоставленном контексте.

### Схема полного цикла

```
                         ИНДЕКСАЦИЯ (однократно)
                         ======================

  admin/knowledge/*.md
         |
         v
  +------------------+
  | Чтение файлов    |
  +------------------+
         |
         v
  +------------------+     chunk_size=300
  | Чанкинг          |     chunk_overlap=100
  | (RecursiveChar-   |     разделители: \n\n -> \n -> пробел -> символ
  |  TextSplitter)   |
  +------------------+
         |
         v
  +------------------+
  | Эмбеддинги       |     модель: BGE-M3 (Ollama)
  | (батч)           |
  +------------------+
         |
         v
  +------------------+
  | Qdrant           |     коллекция: info_chunks
  | (upsert)         |     метрика: cosine
  +------------------+


                      ПОИСК + ГЕНЕРАЦИЯ (на каждый запрос)
                      ====================================

  Вопрос пользователя
         |
         v
  +------------------+
  | Эмбеддинг        |     модель: BGE-M3 (Ollama)
  | вопроса          |
  +------------------+
         |
         v
  +------------------+     top_k=3
  | Поиск в Qdrant   |     similarity_threshold=0.3
  +------------------+
         |
     найдено?
    /        \
  нет        да
   |          |
   v          v
  "Информация  +------------------+
  не найдена"  | Формирование     |
               | промпта          |
               | (контекст +      |
               |  системный +     |
               |  вопрос)         |
               +------------------+
                        |
                        v
               +------------------+
               | Генерация        |     модель: Qwen 2.5 7B (Ollama)
               | ответа (LLM)    |
               +------------------+
                        |
                        v
                  Ответ пользователю
```

## Используемые модели

| Модель      | Назначение | Описание                                                             |
|-------------|------------|----------------------------------------------------------------------|
| BGE-M3      | Embeddings | Мультиязычная модель эмбеддингов с поддержкой русского языка         |
| Qwen 2.5 7B | LLM        | Языковая модель для генерации ответов на основе контекста            |

Обе модели запускаются локально через Ollama и скачиваются автоматически при первом старте.

## Конфигурация

Все параметры задаются через переменные окружения с префиксом `RAG_`:

| Переменная                | По умолчанию            | Сервис | Описание                               |
|---------------------------|-------------------------|--------|----------------------------------------|
| `RAG_OLLAMA_BASE_URL`     | `http://ollama:11434`   | оба    | URL Ollama-сервера                     |
| `RAG_QDRANT_HOST`         | `qdrant`                | оба    | Хост Qdrant                            |
| `RAG_QDRANT_PORT`         | `6333`                  | оба    | Порт Qdrant                            |
| `RAG_EMBEDDING_MODEL`     | `bge-m3`                | оба    | Модель для эмбеддингов                 |
| `RAG_LLM_MODEL`           | `qwen2.5:7b`            | оба    | Модель для генерации ответов           |
| `RAG_COLLECTION_NAME`     | `info_chunks`           | оба    | Имя коллекции в Qdrant                 |
| `RAG_CHUNK_SIZE`          | `300`                   | admin  | Размер чанка (символы)                 |
| `RAG_CHUNK_OVERLAP`       | `100`                   | admin  | Перекрытие между чанками (символы)     |
| `RAG_KNOWLEDGE_DIR`       | `/app/knowledge`        | admin  | Путь к директории с `.md`-файлами      |
| `RAG_TOP_K`               | `3`                     | query  | Количество возвращаемых чанков         |
| `RAG_SIMILARITY_THRESHOLD`| `0.3`                   | query  | Минимальный порог косинусного сходства |

## Добавление данных в базу знаний

1. Поместите `.md`-файл в директорию `admin/knowledge/`.
2. Вызовите `POST http://localhost:8001/ingest`.
3. Коллекция будет полностью пересоздана с учётом всех файлов в директории.
